\n\n--- FILE: ./Reports/Agent_Eval_Benchmarking_Results.md ---\n\n# Agent Evaluation & Benchmarking Results

**Date:** 2026-05-25
**Hardware Used:** Workstation (RTX 5080) & SRE Node (RTX 3070 Ti)

This report summarizes the results of our foundational experiments using `agent-eval`. It covers both **Vision Model Analysis** and **Distributed MAS Swarm Capacity** tests.

## Model Evaluation Summary

| Model | Hardware / Engine | Latency (Avg) | Accuracy / Behavior |
| :--- | :--- | :--- | :--- |
| **`llava:latest` (7B)** | Workstation (Ollama) | Fast | **The Baseline Winner.** Extremely consistent, well-balanced verbosity, and accurate. The best all-rounder for general vision tasks on the Workstation. |
| **`llama3.2-vision:11b`** | Workstation (Ollama) | Moderate | **The Over-Explainer.** Highly accurate and capable of brilliant deep reasoning, but severely penalizes itself in automated evaluations due to excessive verbosity. Best reserved for complex analysis tasks where deep detail is desired. |
| **`Qwen/Qwen2-VL-2B-Instruct`** | SRE Node (vLLM) | Ultra-Fast (~0.6s) | **The Bounding Box Specialist.** Operates blazing fast on the 3070 Ti. Perfect at simple identification (e.g., flowers). However, its heavy grounding training causes it to output literal bounding box coordinates `(101,66)` instead of verbal descriptions for complex architectural scenes unless strictly prompted. |
| **`llava:13b`** | Workstation (Ollama) | Slow | Comparable accuracy to the 7B version, but the latency hit makes it inefficient for rapid MAS triage. |
| **`moondream:latest`** | Workstation (Ollama) | Fast | **Insufficient.** Failed to grasp architectural nuances and lacked the visual reasoning depth required for our use cases. |

## Strategic Takeaways

1. **Model Specialization is Required:** No single model is perfect for every task. `llava` is great for general chats, `llama3.2-vision` is ideal for deep UX/architecture breakdowns, and `Qwen2-VL` is unparalleled for rapid triage and object detection (grounding).
2. **Prompt Engineering:** Models like `Qwen2-VL` require incredibly strict system prompts to constrain their output (e.g., "Do not output coordinates. Describe the scene verbally.") to prevent them from reverting to object detection outputs.

## 2. MAS Swarm Capacity Benchmarks

To validate our Distributed Multi-Agent architecture, we tested the hardware boundaries of both the SRE Node and the Workstation by bypassing the LiteLLM proxy and blasting models directly via the `ws/` and `sre/` routing prefixes.

### SRE Node (RTX 3070 Ti - 8GB VRAM)
Tested using `qwen2.5-coder:7b`.
- **Max Queue Tolerance:** Successfully queued and processed **100 concurrent agents** sequentially without dropping a single connection (Avg Latency: 280s).
- **Context-Heavy OOM Test:** Pushed 25 concurrent agents with 8,000-token payloads. The GPU **did not OOM crash**. Ollama managed the KV Cache gracefully, completing all 25 huge requests in ~60 seconds.
- **Critical Discovery:** Systemd provisioners (e.g., `homelab-inference.service`) will stealth-restart `vLLM` if stopped manually. This steals 6.5GB of VRAM and causes 10-minute timeouts in Ollama. Always stop the systemd service via the toggle script before testing!

### Workstation (RTX 5080 - 16GB VRAM)
Tested using `llama3.2-vision:11b`.
- **Latency Baseline:** Cleared a 25-concurrency swarm in **5.32 seconds total**. The 5080 handles parallel inference nearly 12x faster than the 3070 Ti, even when running a much larger 11B parameter model.

## Proposed Future Experiments

Now that basic classification and understanding are validated, we should expand `agent-eval` to test the models against the following advanced scenarios:

1. **OCR & Structured Extraction:** 
   * **Task:** Pass a messy receipt, a terminal stack trace screenshot, or a form into the models and demand a strict JSON schema output.
   * **Goal:** Verify if smaller models (like Qwen-2B) drop fields compared to heavy models (Llama 3.2 11B).
2. **Spatial & Layout Reasoning:**
   * **Task:** Feed the models a UI mockup or an architectural diagram and ask them to reconstruct the layout or describe the DOM structure.
   * **Goal:** Test their ability to comprehend relationships between visual elements rather than just identifying them.
3. **Multi-Image Comparative Analysis (Anomaly Detection):**
   * **Task:** Send a "before" and "after" image of a dashboard (e.g., a Grafana spike).
   * **Goal:** Determine if models can detect the anomaly and explain what changed across frames (critical for the SRE Watchdog persona).
\n\n--- FILE: ./Reports/MAS_Vision_Offload_Session.md ---\n\n# MAS Vision Offload Session Report

**Date:** 2026-05-25
**Topic:** Architectural Insights on Inference Providers (vLLM vs Ollama)

## Executive Summary
During the process of evaluating vision models (specifically `llama3.2-vision:11b` and `Qwen2-VL-7B`), we attempted to deploy a Tier 2 Vision Watchdog onto the SRE Node (`192.168.99.178`, RTX 3070 Ti) using `vLLM`.

This deployment brought an architectural constraint to light: **vLLM requires static loading**. 

## Architectural Observations

1. **Static vs. Dynamic Loading:**
   * **vLLM (SRE Node):** Models are defined statically at container startup (e.g., via `--model` argument). Changing a model requires a full container restart, resulting in downtime and latency while the model is pulled, loaded into VRAM, and the computation graphs are compiled.
   * **Ollama (Workstation):** Models are dynamically loaded on-demand via API calls. This allows seamless multiplexing and swapping of weights in VRAM without container restarts.

2. **Impact on Multi-Agent Systems (MAS):**
   * **Production/Agent Environments:** The static nature of `vLLM` is highly advantageous. It guarantees that the required weights are persistently pinned in VRAM, eliminating "cold start" latency and preventing race conditions where agents might attempt to swap models out from under each other.
   * **Experimental Environments:** For testing and benchmarking a wide array of models rapidly, `vLLM` is a severe bottleneck. The cycle time to test a new quant involves modifying Docker configuration, restarting services, and waiting for static initialization.

## Strategic Recommendations

For the broader Homelab architecture, we should establish clear boundaries based on these traits:

- **The Workstation (RTX 5080) running Ollama** should remain the primary **Experimental & R&D Hub**. Its dynamic capabilities allow us to rapidly pull and test multiple vision models without friction.
- **The SRE Node (RTX 3070 Ti) running vLLM** should be restricted to **Production/Agent Workloads**. When an experiment "graduates," its chosen model should be statically deployed to the SRE Node. This ensures agents relying on that capability experience 0-second cold starts and highly optimized, batched throughput. 

### SRE Node "Experiment Mode" Consideration
While the SRE node is primarily for production, we should consider supporting an **"Experiment Mode"**. This would involve an infrastructure toggle to gracefully spin down the `vLLM` container, freeing the 8GB VRAM entirely for temporary experimental workloads. 
* **Implementation Path:** `LiteLLM` itself cannot orchestrate Docker container lifecycles (it is strictly an API router). However, our planned `homelab-orchestrator/async-worker` is the perfect candidate to manage this state transition. An agent could submit an async job to "Enable SRE Experiment Mode," triggering the worker to stop `vLLM`, run the tests, and then spin `vLLM` back up—all while LiteLLM uses fallback routing to route around the downtime.

Moving forward, any new evaluation frameworks (like `agent-eval`) should focus on hitting the Workstation Ollama node unless validating a production deployment candidate, or explicitly requesting the `async-worker` to unlock the SRE node.

### Final Thoughts on Experimentation Friction
The rigid nature of the `vLLM/LiteLLM` stack undeniably introduces significant friction during rapid experimental phases. The pipeline of editing Docker manifests, waiting for containers to restart, and enduring long VRAM allocation loops makes rapid A/B testing of models across hardware tedious. 

For future massive-scale experimentation phases, deploying `Ollama` concurrently on **both** the Workstation and the SRE Node would drastically accelerate our iteration velocity. Ollama's ability to seamlessly pull and hot-swap weights dynamically across the network would allow our `agent-eval` script to run multi-GPU comparison grids in seconds rather than hours, without ever touching an infrastructure manifest.
\n\n--- FILE: ./Reports/agent_status_sre_analysis.md ---\n\n# Agent Status MVP: SRE Analysis & Remediation Plan

This report outlines the findings from the initial run of the `agent-status` MVP tool. It is structured into two parallel tracks: **Infrastructure Remediation** and **Tool Enhancements**, allowing the platform and infra teams to address the identified gaps concurrently.

## Track 1: Infrastructure Remediation
*These tasks involve fixing underlying infrastructure issues and observability gaps discovered by the tool.*

### 1. UPS Monitoring Failure (Critical)
- **Issue**: Both Unraid nodes (Dionysus and Archive) report `UPS status unavailable: GraphQL error: Failed to get UPS data: No UPS data returned from apcaccess`.
- **Impact**: We have zero visibility into battery backup status, leaving the storage arrays vulnerable to sudden power events.
- **Action**: Debug the `apcaccess` daemon on Dionysus and Archive. Ensure the USB communication with the UPS is active and the `apcupsd` service is running.

### 2. Missing Log Collection (High)
- **Issue**: Neither Unraid node is shipping logs to Loki (`logs: not_configured`).
- **Impact**: Troubleshooting container crashes or array errors requires manual SSH access.
- **Action**: Deploy Promtail or Alloy to Dionysus and Archive to stream `/var/log/syslog` and Docker container logs to the central Loki instance (`192.168.99.178:3100`).

### 3. UniFi Remote Syslog (Medium)
- **Issue**: Network devices (UDM, USW, UAPs) currently do not ship logs to our centralized observability stack.
- **Impact**: Missing audit logs and connection events for network diagnostics.
- **Action**: Configure the UniFi Controller's remote logging feature to point to our Loki instance.

---

## Track 2: Tool Enhancements
*These tasks involve improving the `agent_status.py` script to provide more accurate and comprehensive data.*

### 1. Fix Uptime Mapping Bug (Medium)
- **Issue**: The SRE node (`homelab-sre`) reports its uptime as `unknown`.
- **Cause**: Prometheus stores the instance label as `node-exporter` for this node. The script parses the label string directly, failing to map `node-exporter` back to the SRE node's IP (`192.168.99.178`) during the uptime phase.
- **Action**: Update `fetch_node_uptimes()` to use the same label-to-IP mapping logic currently used in `fetch_prometheus_up_ips()`.

### 2. Add Storage Capacity Metrics (High)
- **Issue**: The script collects Unraid CPU, Memory, and Disk Temperatures, but ignores Array Capacity.
- **Cause**: The MVP focuses on general compute metrics.
- **Action**: Expand `UnraidDetails` to include `array_utilization_pct`. Update the MCP call parsing to extract total vs used storage space, providing critical capacity warnings to the LLM.

### 3. Track Client Bandwidth Utilization (Medium)
- **Issue**: The tool gathers Wi-Fi satisfaction and RSSI, but lacks throughput data.
- **Cause**: Throughput metrics are available in the UniFi `stat/sta` payload but are currently dropped during parsing.
- **Action**: Extract `tx_bytes-r` (transmit rate) and `rx_bytes-r` (receive rate) from the client data and map them into the `ClientInfo` model to identify bandwidth-heavy devices.

### 4. Support Network Log Status (Low)
- **Issue**: The script currently hardcodes `logs: n/a` for all `uap`, `usw`, and `udm` device types.
- **Action**: Remove the hardcoded `n/a` override and allow the script to check if the network device's IP exists in the `LOKI_SHIPPING_IPS` config list (once Track 1, Task 3 is complete).
\n\n--- FILE: ./Tools/agent-status/README.md ---\n\n# agent-status

A homelab CLI tool that queries live infrastructure data across the network and produces a structured JSON status report with an LLM-generated summary.

## What it does

- Connects to [`homelab-mcp`](http://192.168.99.178:8083) over SSE to fetch UniFi infrastructure devices and all connected clients
- Queries Prometheus (`:9091`) for live metrics scrape status and node uptimes
- Queries Loki (`:3100`) to check log shipping status per host
- Assembles a structured JSON report covering:
  - **8 infrastructure devices** — servers (Unraid, SRE), gateway, switch, APs — each with `logs / metrics / otel` observability status
  - **All network clients** — wired and wireless stations with hostname, IP, VLAN, uplink, RSSI, satisfaction score, and uptime
- Uses a local LLM (`ollama/qwen2.5-coder:14b` via LiteLLM proxy at `:4000`) to generate a 3-5 sentence executive summary

## Prerequisites

| Requirement | Value |
|---|---|
| Python | ≥ 3.14 |
| Package manager | [`uv`](https://docs.astral.sh/uv/) |
| MCP server | `homelab-mcp` running at `http://192.168.99.178:8083/sse` |
| LiteLLM proxy | Running at `http://192.168.99.178:4000/v1` with `ollama/qwen2.5-coder:14b` |
| Prometheus | Running at `http://192.168.99.178:9091` |
| Loki | Running at `http://192.168.99.178:3100` |

## Running

```bash
cd homelab/Tools/agent-status
uv run agent_status.py
```

Output is printed to stdout as formatted JSON. You can pipe it:

```bash

uv run agent_status.py | jq '.clients | length'
uv run agent_status.py | jq '[.clients[] | select(.connection_type == "wireless")] | sort_by(.satisfaction)'
```

## Observability Status Logic

| Device type | Logs | Metrics | OTel |
|---|---|---|---|
| UniFi APs / Switch / Gateway | `n/a` | `n/a` | `n/a` |
| Unraid nodes (Dionysus, Archive) | `not_configured` | live Prometheus check | `n/a` |
| SRE machine (`192.168.99.178`) | `working` | live Prometheus check | `working` |

Metrics status is determined by querying the live `/api/v1/targets` endpoint — a node is `working` if at least one target for its IP (or a known Docker job like `node-exporter` / `cadvisor`) is healthy.

## Architecture

```
Phase 1  MCP SSE     → get_unifi_devices + get_unifi_clients
Phase 2  Prometheus  → /api/v1/targets (which IPs are scraped and healthy)
Phase 3  Loki        → /loki/api/v1/label/host/values (log shipping check)
Phase 4  Prometheus  → node_boot_time_seconds (real uptime for server nodes)
Phase 5  Python      → deterministic parse into Pydantic models
Phase 6  LLM         → free-text summary only (no schema pressure)
```

The two-phase pattern (data collection deterministic in Python, LLM only for narrative) is the recommended approach for local models. See [`homelab-platform/docs/golden_paths/hybrid_ai_patterns.md`](../../../homelab-platform/docs/golden_paths/hybrid_ai_patterns.md) for the full pattern reference.

Infrastructure lists (Unraid nodes, static nodes, OTel/Loki IPs) must be configured in a `config.yaml` file located in the same directory as the script.

\n\n--- FILE: ./Tools/agent-eval/README.md ---\n\n# Agent-Eval

`agent-eval` is a CLI evaluation framework built for the homelab to benchmark agent performance using DeepEval and litellm. Traces are emitted over OTLP directly to your telemetry backend (e.g. Arize Phoenix & Loki via Grafana Alloy).

## Quickstart

Run experiments locally while hitting the SRE machine proxies!

1. **Activate the local environment:**
   ```bash
   source .venv/bin/activate
   pip install -r Tools/agent-eval/requirements.txt
   ```
2. **Execute an experiment:**
   ```bash
   python Tools/agent-eval/cli.py Tools/agent-eval/experiments/refactor_bash.yaml
   ```

## Hardware Targets & LiteLLM

This homelab utilizes two primary GPU nodes:
- **Workstation (RTX 5080)**: Best for heavier models (e.g., 14B+ parameters).
- **SRE Machine (RTX 3070 Ti)**: Best for lighter models or quick tests (e.g., 1.5B - 8B parameters).

By default, the CLI will use:
- **LiteLLM Proxy:** `http://192.168.99.178:4000/v1` (Running on the SRE Machine)
- **OTLP Trace Exporter:** `http://192.168.99.178:4319/v1/traces`

When hitting LiteLLM, requests are automatically routed to the correct machine based on the model specified in your YAML (e.g., `ollama/llama3.1:8b`). You can override the proxy or telemetry endpoints by setting the `LITELLM_API_BASE` and `OTEL_EXPORTER_OTLP_ENDPOINT` environment variables.

### Fallbacks (If LiteLLM is down)

If LiteLLM is not working or you need to bypass it, you can hit the backend inference engines directly by exporting `LITELLM_API_BASE` and updating the model names in your YAML to remove the provider prefix (e.g., remove `ollama/`):

**Workstation (RTX 5080):**
- **Ollama**: `export LITELLM_API_BASE="http://localhost:11434"`
- **vLLM**: `export LITELLM_API_BASE="http://localhost:8000/v1"`

**SRE Machine (RTX 3070 Ti):**
- **Ollama**: `export LITELLM_API_BASE="http://192.168.99.178:11434"`
- **vLLM**: `export LITELLM_API_BASE="http://192.168.99.178:8000/v1"`

## Writing an Experiment YAML

Create a new YAML file in the `experiments/` directory to define your test criteria:

```yaml
experiment_name: "My Evaluation"
system_prompt: "You are an expert agent..."
models_to_test:
  - "ollama/qwen2.5-coder:1.5b-base"
  - "ollama/llama3.1:8b"
judge_model: "ollama/qwen2.5-coder:14b"
test_cases:
  - name: "Refactor feature X"
    input_file: "path/to/local/file.py"
    task_prompt: "Refactor this file to..."
    expected_output_criteria: "The script must use X and Y."
```
> **Note:** Make sure model names are prefixed (e.g., `ollama/`, `vllm/`) to route correctly through the LiteLLM proxy!

## 🤖 AI Workflow (For Antigravity & LLM Assistants)
When asked to create an experiment or benchmark an agent:
1. **Create the YAML:** Write an `experiment.yaml` file into `Tools/agent-eval/experiments/`.
2. **Execute Locally:** Run `python Tools/agent-eval/cli.py Tools/agent-eval/experiments/<name>.yaml` directly in the chat session's terminal. No SSH required! The defaults automatically target the SRE machine.
3. **Summarize:** Read the CLI's standard output and the JSON results file dropped in `results/`, then summarize the winner for the user!

## 🛠️ OpenTelemetry & Tracing Troubleshooting

If you see the warning `LiteLLM:WARNING: opentelemetry.py - Proxy Server is not installed` when running `agent-eval`, it indicates that the local Python environment does not have the necessary dependencies for local client-side OpenTelemetry logging. While LLM requests are successfully processed and tracked by the remote proxy server, **local spans from the client CLI will be skipped.**

### Option A: Enable Client-Side Tracing (Recommended for local logs)
To generate and export traces from the client script directly to your OTLP backend:
1. **Modify `Tools/agent-eval/requirements.txt`**:
   Change `litellm` to `litellm[proxy]`.
2. **Re-install dependencies**:
   ```bash
   pip install -r Tools/agent-eval/requirements.txt
   ```
3. **Verify**:
   Run `agent-eval` and confirm the warning is resolved and spans are visible at `http://192.168.99.178:4319/v1/traces`.

### Option B: Bypassing Client-Side Tracing
If you only care about tracing on the remote LiteLLM Proxy Server and want to silence the client warning:
* Comment out the success and failure callbacks in `Tools/agent-eval/cli.py`:
  ```python
  # litellm.success_callback = ["otel"]
  # litellm.failure_callback = ["otel"]
  ```
\n\n--- FILE: ./docs/CLUSTER_OPERATIONS.md ---\n\n# Cluster Operations & Compute Constraints

This document formalizes the compute boundaries, hardware constraints, and operational guidelines for deploying AI models and workloads within the homelab ecosystem. Sub-agents must adhere to these limits when spinning up local inference engines (Ollama, vLLM, etc.) or designing background tasks.

## 1. Primary Workstation Node (RTX 5080)
- **Host IP:** `192.168.99.247`
- **VRAM Capacity:** 16GB
- **Hardware Role:** Heavy LLM inference node and primary development environment.
- **Model Constraints:**
  - **Maximum Parameter Size:** 14B models (e.g., Qwen 2.5 14B, Llama 3 8B with massive context).
  - **Context Window:** Can support extended context windows (32k+) for 7B-8B models without spilling to system RAM.
- **Usage Guidelines:** 
  - Exclusively use this node for on-demand coding tasks, complex reasoning, and any task requiring large context windows (like ingesting multiple large repositories).
  - Do not deploy persistent background services here, as they compete with the primary user's desktop experience.

## 2. SRE & Local AI Node (RTX 3070 Ti)
- **Host IP:** `192.168.99.178`
- **VRAM Capacity:** 8GB
- **Hardware Role:** Background Inference Node, Telemetry Aggregation, and Provisioning.
- **Model Constraints:**
  - **Maximum Parameter Size:** 7B - 8B models (e.g., Llama 3 8B, Mistral 7B).
  - **Context Window Cap:** **Strictly capped at 4096 tokens.** Exceeding this will cause Out-Of-Memory (OOM) errors or spill over into system RAM, severely degrading performance and potentially crashing background telemetry services.
- **Usage Guidelines:** 
  - Dedicated to background RAG embeddings, Sentinel health checks, and small-context AI orchestrator fallbacks.
  - All prompt evaluations, cron-triggered summarizations, and background alert analysis must be carefully batched to fit within the 4096 context limit.

## 3. Storage Nodes (Unraid)
- **Dionysus** (`192.168.99.115`): Primary Docker host. No dedicated GPU for AI inference. High CPU/RAM capacity.
- **Archive** (`192.168.99.189`): Backup Node.
- **Usage Guidelines:** These nodes handle persistent storage, media, and lightweight stateless Docker containers. AI inference should not be offloaded to these nodes unless strictly CPU-based and highly asynchronous.
\n\n--- FILE: ./docs/MULTI_AGENT_BENCHMARKS.md ---\n\n# Homelab Multi-Agent Benchmarking Framework

## Overview
This document summarizes the custom, declarative benchmarking framework built to evaluate Local LLMs and Multi-Agent Orchestration pipelines within the homelab environment. 

The goal of this architecture is to provide a "golden path" for reproducible experiments. It leverages **DeepEval** for quantitative grading and Python execution verification, paired with standard **OpenTelemetry (OTLP)** for deep trace observability into **Arize Phoenix** and **Loki**.

## Architecture & Hardware Topology
* **Local Workstation (RTX 5080)**: Serves as the primary execution runner. The `agent-eval` CLI is run from here, leveraging its `.venv` while delegating heavy lifting to the SRE node.
* **SRE Node (RTX 3070ti @ `192.168.99.178`)**: 
  * Hosts the LiteLLM Proxy endpoint (`http://192.168.99.178:4000/v1`) to unify model access (vLLM, Ollama).
  * Hosts the **Grafana Alloy** telemetry ingestion point (`http://192.168.99.178:4319/v1/traces`), which fans out traces.
  * Hosts **Arize Phoenix** (`:6006`) for visual LLM span analysis.
  * Hosts **Loki** (`:3100`) for centralized log storage.

## Key Components

### 1. Declarative Experiments (`Tools/agent-eval/experiments/*.yaml`)
All experiments are defined entirely in YAML. This separates the benchmark logic from the code, allowing quick iteration on prompts and models without modifying the Python runners.
* **`refactor_bash.yaml`**: Tests single-agent capability to refactor Bash to Python.
* **`multi_agent_refactor.yaml`**: Tests the ability of 14B+ models to act as Lead Orchestrators.

### 2. The `agent-eval` CLI (`Tools/agent-eval/cli.py`)
A unified CLI that evaluates standard models against DeepEval constraints:
* Reads `experiments/*.yaml`.
* Uses `litellm` routing to forward generation tasks to the remote SRE proxy.
* Leverages LLM-as-a-Judge (`GEval`) to verify output formatting and error-handling requirements based on the YAML definition.
* Automatically exports execution telemetry via standard `opentelemetry` to the remote Alloy collector.

### 3. "Fan-Out" Telemetry Pipeline
We abandoned proprietary SDKs (like Langfuse) in favor of open standards. 
* The CLI natively instruments spans using standard OpenTelemetry and pushes them to Grafana Alloy.
* Alloy pipelines those traces simultaneously into Arize Phoenix (for nested LLM trace analysis and token cost tracking) and Loki (for text-based querying via LogQL).

## 🤖 AI / Agentic Workflow
If you are an AI assistant (like Antigravity or Claude) tasked with benchmarking:
1. **Define the Test:** Create an `experiment.yaml` in `Tools/agent-eval/experiments/`.
2. **Execute Locally:** Run `python Tools/agent-eval/cli.py <path_to_yaml>`. (The CLI natively defaults to targeting the SRE machine).
3. **Analyze:** Read the CLI's output logs and summarize the scores for the user!
\n\n--- FILE: ./docs/LANGFUSE_V3_POSTMORTEM.md ---\n\n# Langfuse v3 Migration & Evaluation Postmortem

## Objective
The purpose of this experiment was to evaluate upgrading the homelab's Langfuse telemetry stack from v2 to v3 to gain deeper observability into multi-agent workflows, specifically hierarchical trace tracking, and quantitative dataset grading (using `DeepEval`). 

## Architectural Changes Explored
Langfuse v3 completely overhauled its architecture for scalability, introducing complex new dependencies:
1. **ClickHouse**: Used for high-performance ingestion and querying of telemetry data (traces, spans, scores, observations).
2. **Postgres**: Retained solely for core entity storage (Users, Projects, API Keys, Datasets).
3. **Redis & Worker Container**: Telemetry ingestion is now decoupled. The `langfuse-web` API container buffers traces into Redis, and a dedicated `langfuse-worker` container asynchronously drains the queue to insert data into ClickHouse.
4. **MinIO / S3**: Required for object storage and media uploads.

## Technical Findings
- **Python SDK Collisions**: The Langfuse v3 Python SDK aggressively utilizes background `asyncio` threads for batching and flushing telemetry. This severely conflicted with `LiteLLM`'s own asynchronous telemetry hooks when running inside synchronous DeepEval metrics (resulting in `RuntimeError: no running event loop` or `unknown async library`).
  - **Resolution**: We successfully bypassed this by completely removing LiteLLM from the custom evaluator classes and using the raw `openai` Python client for local inference, alongside `asyncio.to_thread` for blocking execution tests.
- **Dataset Constraint Errors**: Re-running scripts against the same Dataset without unique run names triggers a silent Postgres unique constraint error `(dataset_id, project_id, name)` on the server, which aborts the creation of Dataset Runs and orphans the traces.
  - **Resolution**: Generating UUIDs for the `run_name` at runtime solved the collisions.
- **Trace Persistence**: Since ingestion is purely asynchronous, short-lived Python scripts will exit before traces upload to the server. Explicitly calling `langfuse.flush()` at the end of the script is mandatory.

## Conclusion & Decision
While Langfuse v3 is an incredibly powerful platform for enterprise teams, the architectural complexity (requiring Postgres, ClickHouse, MinIO, Redis, a Web container, and a Worker container) is overwhelmingly heavy for a localized homelab environment.

**Action Taken**: The Langfuse stack has been entirely torn down. The homelab will revert to lightweight OpenTelemetry (OTEL) configurations for basic tracing, combined with standalone CLI-driven `DeepEval` scripts for experimentation. Future dashboard evaluations will look toward lighter alternatives such as Arize Phoenix.

\n\n--- FILE: ./Learning/local_llm_benchmarking_learnings.md ---\n\n# Local LLM Benchmarking & Multi-Agent Learnings

This document captures the key hardware and architectural learnings from executing multi-agent pipelines (via `agent-eval`) across a constrained local homelab environment.

## Hardware Setup
- **Workstation:** RTX 5080 (~16GB VRAM)
- **SRE Machine:** RTX 3070 Ti (~8GB VRAM)
- **Routing:** LiteLLM proxy pointing to local Ollama instances.

## Key Learnings

### 1. The VRAM Wall & Thrashing Penalty
Running models that exceed available VRAM results in catastrophic latency penalties without corresponding quality gains. 
- **Example:** Running the `qwen3.5:27b` model on the 16GB Workstation forced heavy CPU/RAM offloading. The experiment took **71 minutes** and achieved a GEval score of `0.9`.
- **Comparison:** Running a smaller `14b` model achieved the exact same `0.9` score in just **24 seconds**. 
- **Takeaway:** For text generation and reasoning tasks, it is almost never worth overflowing VRAM. You achieve better or equal quality by using highly optimized models that fit comfortably within your hardware constraints.

### 2. The Multi-Agent "Sweet Spots"
Different models excel at different roles in a multi-agent pipeline based on their parameter count and quantization:

- **The Drafting Champion (Speed):** `qwen2.5-coder:1.5b-base`
  - When hosted on the SRE machine, this model can pump out V1 drafts instantly. When combined with a 14B Refiner on the Workstation, the entire pipeline finishes in under 30 seconds with excellent quality.
  
- **The Orchestrator (Quality):** `command-r:35b` (4-bit quantized)
  - Despite being 35B, its specific 4-bit quantization allows it to run efficiently on the Workstation. It was the only model to consistently achieve perfect **1.0 GEval scores** across different pipelines. It excels at routing, generating sub-prompts, and combining context.
  
- **The Critic (Logic):** `deepseek-r1:7b` / `14b`
  - Using reasoning models specifically for the *Critic* phase allows them to utilize their native `<think>` tags to logically structure feedback.

### 3. Architecture: The Sequential Mob of Experts
Running multiple large models in parallel (Parallel Fan-Out) is impossible on a 16GB GPU. However, you can achieve the same "Mob of Experts" quality by running the models **sequentially**.
- **The Pattern:** Orchestrator (`command-r`) -> Expert A Pipeline (`qwen`) -> Expert B Pipeline (`qwen`) -> Synthesizer (`command-r`).
- **Why it works:** Ollama dynamically loads and unloads the models from VRAM one at a time. This trades raw speed (taking 7-20 minutes depending on the pipeline size) for unparalleled reasoning quality, making it the perfect pattern for asynchronous background tasks where time is not a constraint.

### 4. GEval and JSON Validation Quirks
- When using deep evaluation libraries (like DeepEval), the **Judge Model** must be flawless at strictly adhering to JSON schemas (e.g. `{"score": X, "reason": "Y"}`).
- **Finding:** While `command-r:35b` is the ultimate creative orchestrator, it frequently breaks JSON validation. `qwen2.5-coder:14b` is the undisputed winner for internal GEval tasks, producing highly structured, mathematically reliable JSON outputs without crashing the pipeline.
\n\n--- FILE: ./Learning/multi-agent-patterns/README.md ---\n\n# Multi-Agent Design Patterns

This directory contains standalone, vanilla Python and Golang implementations of the 8 core Multi-Agent System (MAS) design patterns inspired by the [Developer’s guide to multi-agent patterns in ADK](https://developers.googleblog.com/developers-guide-to-multi-agent-patterns-in-adk/).

## What We Are Building & Learning
Monolithic AI agents don't scale well. As tasks get more complex, a single agent's error rate compounds, resulting in hallucinations and failed logic. Decentralizing responsibilities to specific specialized agents creates modular, testable, and reliable AI systems. 

Here we explore 8 architectures:
1. **Sequential Pipeline:** A deterministic "assembly line" where Agent A passes output to Agent B.
2. **Coordinator/Dispatcher:** A central "concierge" routes a task to the correct specialist sub-agent.
3. **Parallel Fan-Out/Gather:** Spawning multiple agents simultaneously for speed and synthesizing their outputs.
4. **Hierarchical Decomposition:** A "Russian doll" model where parent agents delegate sub-tasks down a tree.
5. **Generator and Critic:** An "editor's desk" loop where an agent generates content and a critic enforces hard checks before passing.
6. **Iterative Refinement:** A qualitative loop where an agent generates a draft, a critic provides notes, and a refiner rewrites it.
7. **Human-in-the-Loop:** A safety net where agents pause execution and wait for human authorization before taking high-stakes actions.
8. **Composite Patterns:** Mixing and matching the above patterns for robust real-world applications.

## Hardware & Models
Based on local homelab benchmarking, these examples are designed to point to a local LiteLLM proxy (`http://192.168.99.178:4000/v1`). We utilize the following models based on hardware constraints:
- **Lead Orchestrator:** `qwen2.5-coder:14b` (Assigned to RTX 5080)
- **Subagents / Parallel Workers:** `qwen2.5-coder:7b` (Assigned to RTX 3070ti)

## Running the Examples
Each pattern is contained in its own isolated directory for both `python/` and `golang/`.
Simply execute the `main` file within a directory to see the pattern in action.

## Concept
A fully asynchronous, multi-agent blog generation tool that takes source material (raw notes, research, or documentation) and a target persona, and produces a highly polished, intelligently synthesized final blog post. 

Instead of relying on a single fast prompt, this tool trades **time for quality**, running entirely in the background (taking 10-20 minutes) to execute complex chain-of-thought criticism and multi-draft synthesis.

## Core Architecture
This POC relies on the **Sequential Mob of Experts** pattern to bypass the hardware limits of local consumer GPUs (Workstation: 16GB VRAM, SRE: 8GB VRAM).

1. **The Orchestrator:** `command-r:35b` ingests the persona and generates 2+ hyper-specific Expert sub-prompts.
2. **Sequential Fan-Out (The Mob):** A generation pipeline (e.g., `phi3.5:3.8b` Generator -> `deepseek-r1:7b` Critic -> `qwen2.5-coder:14b` Refiner) iterates over the sub-prompts sequentially to create completely different drafts.
3. **The Synthesizer:** `command-r:35b` merges the disparate drafts into the ultimate post.

## Required Hacks & Workarounds for Local Deployment
To make this viable on the current homelab setup, several critical hacks were required during the POC. When building the final tool, these must be explicitly handled:

### 1. The Sequential Bottleneck Hack
- **The Issue:** Attempting to run a true "Fan-Out" (spawning multiple agents concurrently) crashes the 16GB VRAM Workstation immediately when using >8B models. The resulting system RAM offloading increases latency by 100x (taking 71+ minutes).
- **The Hack:** The framework explicitly runs the Mob's pipelines **sequentially**. Ollama handles swapping the models in and out of VRAM gracefully. 
- **Future Implication:** The final tool must have an internal queuing mechanism that guarantees models are executed one at a time per GPU.

### 2. The JSON Judge Swap
- **The Issue:** `command-r:35b` is a brilliant creative Orchestrator, but when used as an internal evaluator mapping to strict JSON schemas (like `{"score": X, "reason": "Y"}`), it fails validation and crashes the pipeline.
- **The Hack:** We explicitly decouple creative orchestration from systemic data parsing. `qwen2.5-coder:14b` is hardcoded to act as the JSON/GEval backend due to its flawless schema adherence.
- **Future Implication:** The backend of the writing tool must route all structured data parsing tasks to `qwen` and all natural language/merging tasks to `command-r`.

### 3. The 20-Minute Timeout Overrides
- **The Issue:** Standard HTTP proxies (like LiteLLM) and libraries (like `httpx`) default to 1-minute or 5-minute timeouts. When loading a 35B model into VRAM and asking it to synthesize 4,000 words, the generation takes ~15 minutes, causing premature network timeouts.
- **The Hack:** Hardcoded `timeout=1200` (20 minutes) on all LiteLLM client completion calls.
- **Future Implication:** The final tool must be heavily asynchronous (e.g., Celery/Redis workers) and any HTTP clients hitting Ollama/LiteLLM must have infinite or massive (30m+) timeouts.

### 4. Reasoning Token Stripping
- **The Issue:** When using reasoning models (like `deepseek-r1:7b`) as Critics, their `<think>` blocks bleed into the final markdown output, looking unprofessional in a blog.
- **The Hack:** We implemented a regex strip `re.sub(r'<think>.*?</think>', '', output, flags=re.DOTALL)` immediately after completion generation.
- **Future Implication:** The writing tool must have an output sanitizer that automatically cleanses reasoning tags before handing drafts to the Synthesizer.

This document outlines the top-level sabbatical targets, architecture standards, coding logic preferences, and foundational infrastructure facts governing the homelab ecosystem. It acts as the ultimate truth for Antigravity sub-agents operating within this multi-repo structure.

## 1. Core Engineering Philosophy
- **KISS (Keep It Simple, Stupid):** Avoid over-engineering. If a standard tool, bash script, or simple Go binary solves a problem elegantly, prefer it over a complex distributed system until scaling demands it.
- **Open-Source & Local-First Bias:** Default to open-weight LLMs, local compute, and self-hosted tools. Avoid usage-based cloud APIs where local equivalents exist. Commercial products are restricted to highly justified exceptions (e.g., Unraid OS).
- **Strict Isolation (Multi-Repo Pattern):** The ecosystem is purely multi-repo. No single monolithic codebase. High cohesion within repositories, low coupling between them. All shared logic must be packaged or communicated over standard protocols.

## 2. Top-Level Architectural Constraints
- **Protocol-Bound Interaction:** Changes across domain boundaries must not mutate external repository state directly. Services must communicate over standardized, loosely coupled protocols (JSON-RPC, GraphQL, SSE).
- **Declarative Infrastructure Gate:** Applications cannot magically deploy themselves. Modifications require configuration changes merged into `homelab-platform`, which serves as a declarative, pull-based validation gate.
- **Top-Down Context Loading:** A sub-agent operating in a leaf repository must always ingest `workspaces.json` from the `homelab` orchestrator repo to understand its position before modifying code.
- **AI Orchestration (Dual-Layer Stack):** The homelab AI layer consists of a Gateway (LiteLLM) for routing/fallback and a Frontend (Open WebUI) for mobile access and built-in MCP integration. External agents connect directly to MCP servers via Streamable HTTP (SSE) bypassing proxy layers.
- **Infrastructure Execution Boundary:** The actual homelab infrastructure (e.g. Postgres, Redis, vLLM, LiteLLM) is hosted on the SRE machine at `192.168.99.178`. Local docker containers on the workstation should NEVER be spun up for these core services. ## 3. CI/CD Pipeline & Runner Topology
Due to the constraints of personal GitHub accounts, self-hosted GitHub Actions runners cannot be shared globally across multiple repositories.
